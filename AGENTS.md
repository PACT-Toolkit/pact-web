## Commands

```bash
# Development
pnpm run dev:mock       # Start dev server with MSW mocked data (most common)
pnpm run dev            # Start dev server against real backend (requires Vercel link)
pnpm run dev:oauth      # Start dev server with OAuth enabled

# Testing
pnpm run test           # Run unit tests (Vitest)
pnpm run test:watch     # Run unit tests in watch mode
pnpm run pw:open        # Open Playwright E2E UI
pnpm run pw:run         # Run E2E tests headless

# Linting & type-checking
pnpm run lint           # TypeScript check + ESLint (run both before committing)
pnpm run lint:typescript
pnpm run lint:eslint

# Code generation
pnpm api:update         # Fetch swagger specs + regenerate all REST hooks + GraphQL types
pnpm gql:codegen:update # Regenerate GraphQL types only (after modifying .graphql files)
pnpm rest:codegen       # Regenerate REST hooks only
```

To run a single Vitest test file: `TZ=CET vitest run src/app/my_feature/test/my_test.test.ts`

## Skills

Skills live in `.agents/skills/`. Each tool accesses them via its own symlink (e.g. `.cursor/skills/`, `.claude/skills/`).

When creating new skills, **always create them in `.agents/skills/{skill-name}/SKILL.md`**.

Available skills:

- `graphql-to-rest-migration` — step-by-step migration of a feature from GraphQL/Apollo to REST/Swagger (Orval + SWR)
- `handling-rest-challenges` — wire up HTTP 418 challenge flows from REST endpoints using the shared `OutOfBandChallengeDialog`
- `linear-plan` — scope and break down a feature into Linear tasks (planning, ticket creation, project setup)
- `next-best-practices` — Next.js conventions: file/route layout, RSC boundaries, async APIs, metadata, route handlers, image/font optimization
- `translate` — add or update i18next translations across da/en/no/sv
- `vercel-react-best-practices` — React/Next.js performance guidelines from Vercel Engineering (write/review/refactor)
- `web-design-guidelines` — review UI code against the Web Interface Guidelines (a11y, UX, design polish)
- `writing-e2e-tests` — write Playwright E2E tests using standard APIs (no custom helpers)
- `writing-for-smb` — user-facing copy guidelines for small-business and entrepreneur audiences
- `writing-unit-tests` — write Vitest unit tests using standard APIs (no custom helpers)

> Skills are kept in sync across tools by symlinking — `.cursor/skills` and `.claude/skills` both point at `.agents/skills`. To list what is actually installed locally, run `ls .agents/skills`.

## Architecture

Terra is a Next.js 16 app (React 18) using **pnpm** as the package manager.

### Module Hierarchy (ESLint-enforced)

```
pages → app → framework
              ↓
           contexts
```

| From        | Can Import                     |
| ----------- | ------------------------------ |
| `app`       | `app`, `contexts`, `framework` |
| `contexts`  | `contexts`, `framework`        |
| `framework` | `framework` only               |
| `pages`     | `app` only                     |

**Forbidden:**

- `contexts` → `app` (contexts cannot depend on features)
- `framework` → `app`, `contexts` (framework is the lowest level)
- `app` → `pages` (features don't know about routing)

### Path Aliases

Always use the `@/` alias for absolute imports (maps to repo root).
Example: `import { foo } from '@/src/app/feature/domain/foo'`

## Feature Folder Structure

Every feature lives in `src/app/{feature_name}/`:

```
src/app/{feature}/
├── data/                    # Data layer
│   ├── *.graphql            # GraphQL queries/mutations
│   └── __codegen__/         # Auto-generated hooks (DO NOT EDIT)
├── domain/                  # Business logic
│   ├── *_context.tsx        # React contexts
│   └── *_validation_schema.ts
├── mock/                    # MSW mocking
│   ├── data/                # Mock data factories
│   └── handlers/            # MSW request handlers
├── test/                    # E2E tests (Playwright) & unit tests (Vitest)
│   └── *.spec.ts / *.test.ts
├── ui/                      # React components
│   └── {sub_feature}/       # Nested by sub-feature
└── index.ts                 # Barrel exports (public API)
```

## Data Layer

- **GraphQL (legacy):** Write `.graphql` files in `data/` → run `pnpm gql:codegen:update` → import generated hooks from `data/__codegen__/`. Uses Apollo Client.
- **REST (new):** Hooks generated by Orval into `src/__codegen__/rest/{service}/`. Uses SWR. Import from there — never edit generated files.
- **Active migration:** Gradually migrating features from GraphQL to REST/Swagger through 2026. See `README.md` for the three-phase migration workflow and the `graphql-to-rest-migration` skill for step-by-step guidance.

### Adding a new REST service

1. Create `schema/{service}/services.config.yaml` and `schema/{service}/swagger.yaml`.
2. Run `pnpm api:update` to fetch the spec and generate hooks in `src/__codegen__/rest/{service}/`.

**Schema folder naming:** The folder name is used verbatim as the proxy path segment — `/api/lunar/{folder-name}/...` — so it must match the **backend URL path**, not the GitHub repo name. Repo names often include a `-service` suffix that is absent from the actual backend path. Confirm the correct name via the swagger `basePath` or a real mobile network request.

## Conventions

### Naming

| Type          | Convention                     | Example                |
| ------------- | ------------------------------ | ---------------------- |
| Components    | PascalCase with feature prefix | `DirectDebitSideSheet` |
| Hook files    | snake*case with `use*` prefix  | `use_get_setup.ts`     |
| Hook exports  | camelCase                      | `useGetSetup`          |
| Test files    | snake_case with `.spec.ts`     | `direct_debit.spec.ts` |
| GraphQL files | PascalCase                     | `DirectDebit.graphql`  |

Use `uuidv4()` from the `uuid` package for all UUID generation. Import as: `import { v4 as uuidv4 } from 'uuid'`.

### Styling

Use `styled-components` with theme tokens (`theme.backgroundPrimary`, `theme.tokens.spaces[200]`).

### UI Components

Use `@lunarway/matter-web` for all UI components. Before using any component, query the `matter-web-storybook` MCP tools to check available props and usage patterns:

- `list-all-documentation` — browse all available components
- `get-documentation` — get props, types, and example stories for a specific component
- `get-documentation-for-story` — get detailed usage examples for a specific story variant

**Never guess component props** — always check via MCP first.

Use Radix primitives for all new components. React Aria is legacy — do not use it for new work.

### Feature Toggles

Use `useFeatureToggleAnyMatch(['terra-my-feature'])` or `withFeatureToggleAnyMatch` from `@/src/app/features`.

When wrapping a component with `withFeatureToggleAnyMatch`, prefix the inner component with `Base`:

```tsx
const BaseMyView = () => { ... };
export const MyView = withFeatureToggleAnyMatch(['terra-my-feature'])(BaseMyView);
```

Do **not** use `_` prefix (e.g., `_MyView`) — it breaks `react-hooks/rules-of-hooks` because React requires component names to start with an uppercase letter.

## Worktrees

- **Naming:** `../terra-web-{identifier}` (e.g., `../terra-web-atl-123`)
- **After creating:** run `pnpm install` and `pnpm run msw:init` in the new worktree
- **Parallel Playwright:** Set `PLAYWRIGHT_PORT=<unique-port>` (e.g., `PLAYWRIGHT_PORT=3001 pnpm pw:run`) to avoid port conflicts when running E2E tests in multiple worktrees simultaneously

## Gotchas

- **Never edit** `__codegen__/` directories — regenerate with `pnpm gql:codegen:update` or `pnpm rest:codegen`
- Always use `@/` alias for absolute imports — ESLint boundaries will fail on incorrect cross-module imports
- **Never replace `<>` fragments** with wrapper elements (e.g., `<div>`) — grid and flex layouts depend on direct child relationships, and adding a wrapper breaks them
- After cloning or creating a worktree, run `pnpm run msw:init` once to initialize the MSW service worker
